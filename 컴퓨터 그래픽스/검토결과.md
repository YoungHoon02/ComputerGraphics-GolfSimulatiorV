## 1. Core/SurfaceType.cs — ⚠️ 부분 충족

**명세 요구**: `enum + static class`. enum(표면 종류) + static class가 `GetFriction()`, `GetRestitution()`, `GetTangentialFriction()` 제공, 표면별 파라미터 표(μ/e) 보유. *"Physics Material 사용 금지 → 코드에서 직접 관리"*.

**실제 구현**:
```csharp
public enum SurfaceType
{ None, Fairway, Rough, Bunker, Green, OutOfBounds, Obstacle }
```
- enum **만** 존재. 명세가 요구한 static class(파라미터 표 + 3개 메서드)가 이 파일에 **없음**.

**판정 근거**:
- ⚠️ enum 자체는 존재하며, 오히려 명세 표(5종)보다 `None`, `OutOfBounds`를 추가해 OB 처리에 유리(➕ 합리적 확장).
- ❌ **static class 미구현**: `GetFriction/GetRestitution/GetTangentialFriction` 및 μ/e 파라미터 표가 `SurfaceType.cs`에 없음.
- ⚠️ 대신 마찰/반발 로직이 `BallPhysics.cs` 내부 **private 메서드**(`GetFrictionAccel`, `GetRestitution`)로 구현됨 → 기능은 존재하나 명세가 의도한 **공용 static 분리**가 아님. 명세 2-2의 설계 의도(BallPhysics·Predictor가 공유)와 어긋날 수 있음. → BallPhysics/Predictor 검토 시 교차 확인 필요.

**확인 필요(메모)**: `GetTangentialFriction`(접선 마찰 f) 대응 로직이 BallPhysics에 실제로 있는지 → 3번 항목에서 확인.

## 2. Ball/PhysicsCore.cs — ⚠️ 부분 충족 (메서드 4개 중 2개)

**위치 차이**: 명세는 `Scripts/Core/`, 실제는 `Scripts/Ball/`. ❌ 경로 불일치(기능엔 무관하나 명세 구조와 다름).

**명세 요구 메서드 4종 대비**:

| 명세 메서드 | 구현 | 비고 |
|---|---|---|
| `Step(ref pos, ref vel, externalAccel, dt, gravity)` | ⚠️ | `Step(ref pos, ref vel, acceleration, dt)` — **gravity 파라미터 없음**. 중력을 호출부가 acceleration에 합쳐 넘기는 구조. 적분 자체는 정확. |
| `ReflectWithRestitution(v, n, e, f)` | ❌ | PhysicsCore에 **없음**. 반발 로직이 `BallPhysics.cs` private `GetRestitution`로 분산됨. |
| `CalculateLaunchVelocity(power, pitch, forward)` | ✅ | 구현됨. XZ 평면 투영 후 pitch 적용 → 정상. |
| `IsStopped(velocity, threshold)` | ❌ | PhysicsCore에 **없음**. 정지 판정이 BallPhysics의 `stopThreshold`로 분산됨. |

**판정 근거**:
- ✅ **Semi-implicit Euler 정확**: `velocity += a*dt; position += velocity*dt` — 명세 핵심 수식(속도 먼저→위치)과 정확히 일치. 평가 항목 "발사체 운동 15점" 핵심 충족.
- ✅ `CalculateLaunchVelocity`: `ProjectOnPlane`으로 forward를 수평화 후 cos/sin으로 pitch 합성 → 발사 초기 속도 계산 적절.
- ⚠️ `Step` 시그니처가 명세와 다름(gravity 분리 안 됨). 동작은 동일하나, 호출부가 중력을 빠뜨리면 버그 가능 → BallPhysics/Predictor에서 중력 합산 일관성 교차 확인 필요.
- ❌ **공용화 설계 붕괴 위험**: 명세 의도(2-8, Q4)는 *BallPhysics와 TrajectoryPredictor가 PhysicsCore.Step + ReflectWithRestitution을 공유*해 예측선이 실제와 일치하는 것. 그런데 `ReflectWithRestitution`이 PhysicsCore에 없고 BallPhysics private에만 있으면, **Predictor가 반발을 재현 못 해 착지/바운스 예측이 실제와 어긋날 수 있음**. → 8번(TrajectoryPredictor)에서 반드시 확인.

## 3. Ball/BallPhysics.cs — ⚠️ 물리는 충족, 게임 흐름 연동은 ❌ 다수 누락

### 제약사항(5장 평가기준) — ✅ 전부 준수
- ✅ Physics Material 미사용 (반발/마찰을 코드 필드로 직접 관리)
- ✅ `Rigidbody.AddForce()` 미사용 — `velocity` 변수 직접 연산
- ✅ `rb.useGravity = false`, `isKinematic = false` (Awake에서 보장), 중력은 코드에서 `new Vector3(0,-gravity,0)`로 계산
- ✅ `OnCollisionEnter`에서 `collision.GetContact(0).normal` 사용 (contacts[] 배열 미사용)
- ✅ Semi-implicit Euler: `a_total = gravity + wind + friction` → `PhysicsCore.Step()` 호출. **바람 외력(`WindManager.GetWindAccel()`) 정상 합산** (평가 "바람 외력 10점" 충족).

### 충돌 반발 수식 — ✅ 정확
```csharp
Vector3 v_n = Vector3.Dot(velocity, n) * n;   // 법선 성분
Vector3 v_t = velocity - v_n;                 // 접선 성분
velocity = -e * v_n + (1f - f) * v_t;         // 반발 e, 접선마찰 f
```
법선/접선 분리 반발이 물리적으로 옳음(평가 "충돌 반발 15점" 핵심 충족). 단, 명세가 요구한 `PhysicsCore.ReflectWithRestitution()` **공용 메서드가 아니라 인라인 구현** → Predictor 재사용 불가(2번 우려 확정).

### 필드/메서드 명세 대비
| 명세 | 실제 | 판정 |
|---|---|---|
| `Velocity` (public get) | `public Vector3 Velocity => velocity` | ✅ |
| `IsFlying`/`IsGrounded`/`CurrentSurface` | 모두 public get 존재 | ✅ |
| `gravity = 9.81f` | 9.81f | ✅ |
| `stopThreshold = 0.08f` | **0.1f** | ⚠️ 값 다름 |
| `Launch(power,pitch,forward)` | 존재, isFlying=true | ✅ |
| `ResetToPosition(position)` | **`ResetBall(position)`** | ⚠️ 이름 다름 |
| `GetLastValidPosition()` | **없음** | ❌ 미구현 |
| 이벤트 `OnBallStopped` | **없음** | ❌ 미구현 |
| 이벤트 `OnBallOutOfBounds` | **없음** | ❌ 미구현 |

### 🔴 핵심 문제 — 게임 흐름 연동 부재 (통합 테스트 영향)
1. **이벤트 2개(`OnBallStopped`/`OnBallOutOfBounds`) 미구현** → 명세상 GameManager가 구독해 "정지→다음 타수", "OB→벌타+1"을 처리. 이벤트가 없으니 **GameManager/ScoreManager로 신호가 전달되지 않음**. → 9번 GameManager에서 연결 여부 교차 확인.
2. **OB 처리를 BallPhysics가 자체적으로** `ResetBall(startPosition)` + `Debug.Log("Out of bounds")`로 끝냄 → **벌타(+1) 누락**(평가/체크리스트 "OB→벌타+1" 깨질 가능성 ❌).
3. **홀인 처리 중복**: `OnTriggerEnter`에서 `"Hole"` 태그를 BallPhysics가 직접 `Stop()`+`Debug.Log("Hole in")` 처리 → 명세상 별도 `HoleTrigger.cs`(HoleCup) 책임과 **중복**. ScoreManager 클리어 처리로 연결 안 됨 ❌. → 4번 HoleTrigger와 교차 확인.

### 마찰 파라미터 — ⚠️ 명세 표와 불일치
- `GetSurfaceFriction()` 값: fairway **0.85**, rough **1.4**, bunker **2.2**, green **0.45**.
  명세 μ표(green 0.05 / fairway 0.15 / rough 0.35 / bunker 0.70)와 **수치 전혀 다름**(게임플레이 튜닝값으로 보임). 마찰 가속 `-mu*gravity*v.normalized`(속도 무관 등감속 모델) → 명세의 μ 의미와 모델이 다름.
- `Obstacle` 표면 마찰 case 없음 → default(fairway)로 처리 ⚠️.
- **접선 마찰 `f`가 단일 전역값(0.1f)** → 명세 `GetTangentialFriction(surface)` 표면별 분리 아님 ❌ (1번 메모 확정).

### 기타
- ⚠️ `Update()`에 `R키 리셋` 임시 테스트 코드 존재 → 명세상 R은 InputManager 담당. 입력 중복(6번 교차 확인).
- ⚠️ 관통 보정 `n * 0.01f` (명세 0.005f), 표면 판정은 Raycast+충돌 하이브리드(명세는 충돌 기반) — 합리적 변형이나 명세와 차이.
- ➕ `LayerToSurface`에 오탈자 레이어명 `"Fairaway"` 허용 케이스(방어적).

**종합**: 물리 수식·제약 준수는 우수하나, **명세가 의도한 이벤트 기반 게임 흐름 연동이 통째로 빠져 있어** 단독 플레이 흐름(타수/벌타/클리어)이 정상 동작하지 않을 가능성이 큼. GameManager(9)·HoleTrigger(4)·ScoreManager(10) 검토 시 연결 단절을 집중 확인.

| HoleCup 부착, Trigger | `[RequireComponent(typeof(Collider))]` | ✅ |
